# SubagentStop Event Payload Shape

## Hypothesis
SubagentStop event stdin provides session_id, agent_type, agent_id fields

## Method

1. Searched `hooks/` directory in the deepflow repo for all existing hook JS files and studied their stdin JSON parsing patterns
2. Read `df-statusline.js`, `df-tool-usage.js`, `df-execution-history.js`, `df-dashboard-push.js` to understand the common hook input shape
3. Searched `~/.claude/cache/changelog.md` for all `SubagentStop`, `agent_type`, `agent_id`, and related field additions across versions
4. Examined `~/.claude/plugins/marketplaces/claude-plugins-official/plugins/plugin-dev/skills/hook-development/SKILL.md` for the canonical hook input format documentation
5. Read `~/.claude/plugins/marketplaces/claude-plugins-official/plugins/plugin-dev/skills/hook-development/scripts/test-hook.sh` for the official sample payload for Stop/SubagentStop
6. Searched existing spec `specs/doing-subagent-instrumentation.md` for constraint notes about missing fields
7. Searched for `agent_type` enum values across `~/.claude/` session transcripts and plugin docs

## Findings

### Common Fields (all hook events)
All hooks receive these fields via stdin JSON (confirmed by `df-tool-usage.js`, `df-execution-history.js`, and SKILL.md):

```json
{
  "session_id": "abc123",
  "transcript_path": "/path/to/transcript.txt",
  "cwd": "/current/working/dir",
  "permission_mode": "ask|allow",
  "hook_event_name": "SubagentStop"
}
```

### SubagentStop-Specific Fields

From changelog analysis:

| Field | Added In | Source |
|-------|----------|--------|
| `reason` | v1.0.41 | Split from Stop hook; Stop/SubagentStop share `reason` field |
| `hook_event_name` | v1.0.41 | Added to all hook inputs |
| `agent_id` | v2.0.42 | "Added `agent_id` and `agent_transcript_path` fields to `SubagentStop` hooks" |
| `agent_transcript_path` | v2.0.42 | Same changelog entry as `agent_id` |
| `agent_type` | v2.1.69 | "Added `agent_id` (for subagents) and `agent_type` (for subagents and `--agent`) to hook events" |
| `last_assistant_message` | post-v2.0.42 | "Added `last_assistant_message` field to Stop and SubagentStop hook inputs" |

### Full SubagentStop Payload Shape (current)

```json
{
  "session_id": "string — the subagent's own session ID",
  "transcript_path": "string — path to subagent's transcript file",
  "cwd": "string — working directory",
  "permission_mode": "string — ask|allow|...",
  "hook_event_name": "SubagentStop",
  "reason": "string — why the subagent is stopping",
  "agent_id": "string — unique identifier for this agent instance",
  "agent_transcript_path": "string — path to agent transcript (same as transcript_path or subagent-specific)",
  "agent_type": "string — e.g. 'Explore', 'general-purpose', or agent name for --agent sessions",
  "last_assistant_message": "string — final text response from the subagent"
}
```

### agent_type Values
From session transcripts and plugin docs, observed values include:
- `"Explore"` — used with `subagent_type: "Explore"` in Task tool calls
- `"general-purpose"` — used with `subagent_type: "general-purpose"` in Task tool calls
- Named agent values (e.g. `"reasoner"`) — when spawning a named `.claude/agents/` agent via `--agent`

The `agent_type` field reflects the `subagent_type` argument passed to the Task tool, or the agent name for `--agent` invocations.

### Critical Constraint: No parent_session_id
From `specs/doing-subagent-instrumentation.md`:
> `SubagentStop` does NOT include `parent_session_id` (GitHub Issue #19448) — classification is by presence in registry, not by parent linkage

This means **you cannot reliably identify which orchestrator spawned a subagent** from the SubagentStop payload alone. The registry approach (append `session_id` → used to classify sessions later) is correct.

### session_id Semantics
From `specs/doing-subagent-instrumentation.md`:
> Sequential subagents may share session_id (GitHub Issue #7881)

This means multiple sequential subagents may produce SubagentStop events with the same `session_id`. The registry must store all entries; the parser handles the `'mixed'` case.

## Conclusion

**confirmed** — SubagentStop event stdin provides `session_id`, `agent_type`, and `agent_id` fields, matching the hypothesis.

Summary of verified fields needed for the registry:
- `session_id` (string): present since v1.0.41 as a common field
- `agent_type` (string): added in v2.1.69, reflects the subagent_type or agent name
- `agent_id` (string): added in v2.0.42, unique instance identifier
- `timestamp`: NOT in the payload — hook must generate this itself with `new Date().toISOString()`

The target JSONL record `{"session_id":"abc-123","agent_type":"Explore","agent_id":"agent_1","timestamp":"..."}` is achievable: three fields come from stdin, timestamp is generated by the hook.
